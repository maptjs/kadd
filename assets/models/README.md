# نموذج تصنيف سجادة الصلاة — خطة العمل

النموذج غير موجود بعد. هذا أكبر عائق تقني في المشروع كامل. الخطة:

## 1. جمع البيانات (Dataset)
- **الفئة الإيجابية (سجادة صلاة):** صوّر عشرات السجادات المختلفة (أنماط، ألوان، أحجام)
  من زوايا وإضاءات متعددة. الهدف الأولي: 300-500 صورة كحد أدنى للانطلاق، مع تنويع:
  - إضاءة نهارية/مسائية/صناعية
  - السجادة مفرودة بالكامل / مطوية جزئيًا / بزاوية
  - أرضيات خلفية مختلفة (بلاط، سجاد، خشب)
- **الفئة السلبية (كل ما هو غير سجادة صلاة):** صور عشوائية لغرف، أثاث، سجاد عادي غير
  مخصص للصلاة، أرضيات فارغة. يجب أن تكون الفئة السلبية متنوعة جدًا حتى لا يتعلم
  النموذج فرقًا سطحيًا (كاللون فقط).
- فكرة عملية: اجمع مساهمات من مستخدمين تجريبيين (beta testers) عبر نموذج بسيط
  لرفع الصور، بدل تصويرها كلها بنفسك.

## 2. التدريب (Transfer Learning)
- استعمل نموذج مُدرَّب مسبقًا خفيف (MobileNetV2 أو EfficientNet-Lite) كقاعدة،
  وأعد تدريب الطبقات الأخيرة فقط (transfer learning) على بيانات السجادة —
  هذا يحتاج بيانات أقل بكثير من التدريب من الصفر.
- أدوات مقترحة: TensorFlow / Keras لتدريب النموذج، ثم تحويله إلى
  `.tflite` عبر TensorFlow Lite Converter، مع quantization (int8) لتصغير
  حجمه وتسريعه على الهاتف.
- بديل أسرع للانطلاق: Google's Teachable Machine (teachablemachine.withgoogle.com)
  يصدّر مباشرة نموذج TFLite من صور تُرفع عبر المتصفح — مفيد لنسخة أولى (MVP)
  قبل الاستثمار في خط تدريب مخصص.

## 3. الدمج مع Flutter

**منجز بالفعل** — `lib/services/rug_classifier.dart` جاهز ويشتغل بمجرد ما
تضيف الملفين التاليين لهذا المجلد (`assets/models/`):

- `rug_classifier.tflite` — النموذج نفسه
- `labels.txt` — حاليًا فيه محتوى وهمي (placeholder) بصيغة Teachable Machine
  القياسية (`0 rug` / `1 not_rug`) — استبدله بالملف الحقيقي، مع الحفاظ على
  أن تحتوي إحدى التسميات على كلمة "rug" (بأي حالة أحرف).

**ملاحظة تقنية:** الكود يستعمل `google_mlkit_image_labeling` (خيار
LocalLabelerOptions لنموذج مخصص محلي) وليس حزمة `tflite_flutter` مباشرة —
هذه الأخيرة فيها علة غير محلولة حاليًا مع إصدارات Dart SDK الحديثة (راجع
التعليق في `pubspec.yaml`). محرك ML Kit الأصلي يتعامل تلقائيًا مع صيغة
النموذج (float32 أو quantized) داخليًا، فلا داعي لأي معالجة يدوية للصورة
أو التنسورات من جانبنا.

`RugScanScreen` يحمّل النموذج مرة واحدة فـ `initState` ويستدعي
`_classifier.classify(imagePath)` بمسار الصورة الملتقطة مباشرة (ML Kit
يتطلب مسار ملف حقيقي، ليس بايتات فالذاكرة). إذا لم يوجد
`rug_classifier.tflite` بعد، `load()` يفشل بصمت والتصنيف يرجّع 0.0 دائمًا.

**ملاحظة Gradle محتملة:** بعض توثيقات ML Kit توصي بإضافة
`aaptOptions { noCompress "tflite" }` (أو ما يعادلها بصيغة Kotlin DSL:
`androidResources { noCompress.add("tflite") }`) داخل
`android/app/build.gradle` (أو `.kts`) حتى لا يُضغط ملف النموذج أثناء
التعبئة فالـ APK بطريقة تفسده. لم أضف هذا تلقائيًا لسكريبتات البناء لأن
صيغة Gradle الدقيقة تختلف حسب نسخة Flutter/AGP عندك بالضبط، وتعديل خاطئ
هنا يكسر البناء كامل. إذا حمّلت نموذجًا حقيقيًا وفشل تحميله فقط على جهازك
(بينما يخدم فـ Teachable Machine مباشرة)، هذا أول شيء تتحقق منه.

## 4. اختبار ميداني
- قبل الإطلاق، اختبر مع مستخدمين حقيقيين بسجاداتهم الخاصة — دقة النموذج على
  بيانات لم يرها أثناء التدريب هي المقياس الحقيقي، ليس دقة بيانات التدريب.
